# Backend check

Runs this repo's full validation loop: Ruff, Alembic migrations, and the pytest suite against a
real PostgreSQL database (this codebase deliberately has no mocked-DB test mode — see
`CLAUDE.md`/`README.md` for why). Use this instead of improvising a validation sequence, since a
couple of steps here aren't discoverable from the error messages alone.

## Why this needs a dedicated skill

Two things about this repo bite you if you guess instead of following this sequence:

1. **Tests need their own database.** `tests/conftest.py` targets `POSTGRES_DB=app_test`, never the
   dev database, and refuses to run against anything else. If `app_test` doesn't exist yet, `pytest`
   fails with a connection error that looks like a config problem but is just a missing database.
2. **Migrations for tests are applied by a synchronous fixture.** This is intentional (Alembic's
   `env.py` drives its own `asyncio.run()`, which conflicts with pytest-asyncio's per-test event
   loop if done async) — don't "fix" a loop-related test failure by changing that fixture's scope
   or making it async without re-reading the comment in `tests/conftest.py::_apply_migrations`.

## Steps

1. **Confirm PostgreSQL is reachable.** Locally that's usually already running; in a fresh sandbox
   you may need to start it (e.g. `service postgresql start` on Debian-based images) and set a
   password for the `postgres` role if one isn't configured yet.

2. **Ensure both databases exist** — the app's own dev DB (from `.env`, default `app`) and the test
   DB:
   ```bash
   createdb app        2>/dev/null || true
   createdb app_test    2>/dev/null || true
   ```
   (Use `PGPASSWORD=... psql -h <host> -U <user> -c "CREATE DATABASE app_test;"` if `createdb` isn't
   on PATH or auth needs a password.)

3. **Install dependencies** if not already in a venv:
   ```bash
   python -m venv .venv && source .venv/bin/activate
   pip install -r requirements-dev.txt
   ```
   Copy `.env.example` to `.env` on first run so `Settings` has something to load.

4. **Lint and format-check** (fast, catches obvious issues before the slower steps):
   ```bash
   ruff check .
   ruff format --check .
   ```
   If either fails on style/import-order only, `ruff check . --fix && ruff format .` before
   re-running — don't hand-fix things Ruff fixes automatically.

5. **Apply migrations to the dev database and check for drift** (this exercises the migration path
   independently of the test suite, which manages its own migrations against `app_test`):
   ```bash
   alembic upgrade head
   alembic check
   ```
   `alembic check` fails if the SQLAlchemy models and the migration files have diverged — usually
   because a model changed without a matching `alembic revision --autogenerate`. If a migration
   touches a Postgres-native enum, remember Alembic's autogenerated `downgrade()` does **not** drop
   the enum type automatically (see the comment in
   `alembic/versions/*_create_users_and_refresh_tokens_tables.py`); a hand-rolled enum
   create/drop is required, or a repeat `upgrade head` after a `downgrade` fails with
   "type already exists".

6. **Run the test suite.** `pytest` alone runs everything; scope it down while iterating on one
   area:
   ```bash
   pytest                                   # full suite
   pytest tests/test_auth.py                # one file
   pytest tests/test_auth.py::test_login_success   # one test
   pytest -k "refresh"                      # by keyword
   ```
   Tests truncate all tables after each run, so re-running is safe and doesn't need manual cleanup
   between attempts.

7. **Report results plainly**: which of lint/migrations/tests passed, and for any failure, the exact
   command and error — not a paraphrase. If `docker compose up --build` is also expected to be
   validated, note explicitly whether it was actually run or whether image pulls were blocked by
   network policy (a real limitation encountered before in sandboxed environments, not a scaffold
   defect) — never claim it passed without having run it.

## When something in this repo changes structurally

If you add a new SQLAlchemy model, confirm it's imported in `app/models/__init__.py` — Alembic's
autogeneration reads `Base.metadata` via `app/db/base.py`, which only sees models that have actually
been imported somewhere. A model that exists but isn't imported silently doesn't show up in
`alembic revision --autogenerate -m "..."`.
